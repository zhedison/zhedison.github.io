关于参数的问题

传递参数的方式有：方法体传参、请求体传参、请求路径传参、请求头传参，其实与controller中的传参方式是差不多的，有很重要的一点是，使用openfeign调用服务需要传参的时候，必须加上传参的类型@RequestParam、@RequestBody、@PathVariable、@RequestHeader。

继承的问题

如果有很多很多个服务都调用同一个服务的接口，那岂不是每个服务都需要写一份一样的代码，而且如果那个被调用的服务改了什么，那么调用这个服务的服务就需要改变，那不是一改就要改好多个？所以，我们可以把调用的方法抽出来，放到一个接口中，这个接口不被谁调用，只用来被继承，那么继承这个接口的服务就拥有了这些方法了，就不用写n多份了。

优缺点分析：

  1. 使用继承特性，代码简洁明了，不易出错，不必担心接口返回值写对没，接口地址写对没，如果接口地址有变化，也不用所有服务 大动干戈，只需要修改 API模块即可。
  2. 前面提到的在 feign-consumer 中绑定接口时，如果是 key/value 形式的参数或者放在 header 中的参数，就必须要使用 @RequestParam 注解或者 @RequestHeader 注解，这个规则在这里一样适用，即在 API 中定义接口时，如果涉及到相关参数，该加的@RequestParam 注解或者 @RequestHeader 注解一个都不能少。
  3. 当然，使用了继承特性也不是没有缺点，继承的方式将 provider 和 feign-consumer 绑定在一起，代码耦合度变高，一变俱变，此时就需要严格的设计规范，否则会牵一发而动全身，增加项目维护的难度。

二、Hystrix

​ 由于微服务是服务与服务之间的调用，那么与单机不同，微服务不可避免的存在网络波动的问题，虽然OpenFeign有自带的重试机制，但是这不够解决我们所有的问题，假设现在有如下场景A调用B，B调用C。现在B调用C重试了好几次都不通，直接报错，那么A调用B肯定也不通了，这就造成了故障蔓延。

​ 因此，为了避免这样的问题发生，有一个降级机制是很必要的，也就是说，当B调用C不通的时候，降级去调用另一个方法，这就不会直接报错而终止服务了。

Hystrix+普通方法

只需要在方法上加上@HystrixCommand注解就可以了（记得加依赖），里面有一个fallbackmethod属性，添加降级的方法即可。

Hystrix除了可以做降级之外，还可以做合并请求，我们知道频繁的发送请求是很耗费时间的，那为了提高效率，就有了合并请求。合并请求将一段时间内所有的请求进行合并（比如说200ms内的请求都合并起来，超过200ms的就下一轮），再统一的进行请求。然后，响应结果再根据请求来进行分配。
