Spring security攻略-02

一、密码加密

1、PasswordEncoder介绍

在Spring Security中与密码加密相关的都是由PasswordEncoder来主导的，这个接口有很多是实现类

其中最常用的就是第一个BCryptPasswordEncoder了，PasswordEncoder接口只有三个方法：

  • encode：用于对密码进行加密
  • matches：用于对密码进行比对
  • upgradeEncoding：用于表示密码是否需要再次加密，比如以前的密码用的是md5加密，但是现在md5加密不安全了，想换成别的加密方式，那么就需要对密码进行更新操作。

那么这里面进行密码校验的PasswordEncoder又是怎么来的呢？

构造方法里面直接就把PasswordEncoder给设置了，意思就是在创建DaoAuthenticationProvider 的时候就已经设置了PasswordEncoder。那么是不是就直接使用这个PasswordEncoder了呢？其实不是的，如果我们自己配置了PasswordEncoder的话，就用我们自己配置的，如果是没有配置的话就用默认的。那么怎么知道是这样的？

3、默认的PasswordEncoder

我们知道如果我们自己没有配置PasswordEncoder的话会默认创建一个PasswordEncoder，那什么是默认的？

看看PasswordEncoderFactories里面的这个方法

返回了一个叫做DelegatingPasswordEncoder的PasswordEncoder，看这个名字像是一个代理的PasswordEncoder，看看他的定义

解析：

  1. DelegatingPasswordEncoder 也是实现了 PasswordEncoder 接口，所以它里边的核心方法也是 两个：encode 方法用来对密码进行编码，matches 方法用来校验密码。
  2. 在 DelegatingPasswordEncoder 的构造方法中，通过 通过传入的两个参数 encodingId 和 encoders ，获取到默认的编码器赋值给 passwordEncoderForEncode，默认的编码器实际上就 是 BCryptPasswordEncoder。
  3. 在 encode 方法中对密码进行编码，但是编码的方式加了前缀，前缀是 {编码器名称} ，例如如果 你使用 BCryptPasswordEncoder 进行编码，那么生成的密码就类似 {bcrypt}$2a$10$oE39aG10kB/rFu2vQeCJTu/V/v4n6DRR0f8WyXRiAYvBpmadoOBE. 。这样有什 么用呢？每种密码加密之后，都会加上一个前缀，这样看到前缀，就知道该密文是使用哪个编码器 生成的了。
  4. 最后 matches 方法的逻辑就很清晰了，先从密文中提取出来前缀，再根据前缀找到对应的 PasswordEncoder，然后再调用 PasswordEncoder 的 matches 方法进行密码比对。
  5. 如果根据提取出来的前缀，找不到对应的 PasswordEncoder，那么就会调用 UnmappedIdPasswordEncoder#matches 方法，进行密码比对，该方法实际上并不会进行密码 比对，而是直接抛出异常。

4、密码加密自动升级（其实是根据你的配置来进行改变）

Spring Security 支持根据你配置的PasswordEncoder来进行密码加密方式的改变，比如说之前是明文的方式存储的，前缀是{noop}，现在要进行bcrypt加密，那一大堆密码在数据库里面，不可能一个个的去改，而且加密过后的密码（比如之前是md5加密）也改不了。解决方法就是等用户再次登录的时候，根据用户登录时输入的密码来进行加密，然后根据用户名来去数据库更新。注意：PasswordEncoder只能用DelegatingPasswordEncoder这个实例，因为只有这个实例是可以识别前缀的。

假设现在我们的数据库中存储的是明文{noop}123，现在我们想将密码加密成bcrypt。需要两步，

  1. 在Service实现UserDetailsPasswordService

updatePassword方法就是给你去数据库更新密码的，参数newPassword就是已经进行过新的加密方式加密的密码，只需要将其更新到数据库就可以

 2.需要在配置类里面配置一个DelegatingPasswordEncoder，里面指定你的加密方式，

这里设置了加密的方式为bcrypt，这样就大功告成了。

二、自动登录（记住我）

1、浅分析一波

自动登录其实就是，登录完一次之后，关闭浏览器，再次访问不需要登录直接可以访问，很多网站都有这样的功能，这样可以避免用户频繁的登录。那么在Spring Security中怎么实现呢，很简单，只需要在过滤器链中加上remenberme的配置就可以了

可以看到登录请求携带的参数多了一个，

remember-me这个字段，表示当我们想自定义登录页面的时候，如果想做这个自动登录的功能，只需要在表单里面添加一个checkbox，key是remember-me，值是on就可以了。

在cookie里面设置了一个remember-me的字段，base64解码之后是这样的

首先第一段是用户名，第二段是两个星期后的时间（表示这个cookie的过期时间），第三段是使用 MD5 散列函数算出来的值，他的明文格式是 username + “:” + tokenExpiryTime + “:” + password + “:” + key ，最后的 key 是一个散列盐值，可以用 来防治令牌被修改。

2、看看源码

看源码主要分两个部分，一个部分是这个cookie是怎样生成出来的，第二部分是服务端是怎么解析我们传进来的那个cookie的。

3、风险

如果我们开启了自动登录，那么返回给浏览器的那个令牌非常重要，只要我们掌握了令牌，就可以不用输入密码登录了。这样，一旦我们的令牌丢失，那么别人就可以随意的使用我们的用户访问，这是非常危险的，那么怎么降低这些风险呢？

a、第一种持久化令牌

持久化令牌就是在基本的自动登录功能基础上，又增加了新的校验参数，来提高系统的安全性，这一些 都是由开发者在后台完成的，对于用户来说，登录体验和普通的自动登录体验是一样的。 在持久化令牌中，新增了两个经过 MD5 散列函数计算的校验参数，一个是 series，另一个是 token。

其中，series 只有当用户在使用用户名/密码登录时，才会生成或者更新，而 token 只要有新的会话， 就会重新生成，这样就可以避免一个用户同时在多端登录，就像手机 QQ ，一个手机上登录了，就会踢 掉另外一个手机的登录，这样用户就会很容易发现账户是否泄漏。

b、二次校验

持久化令牌的方式其实已经安全很多了，但是依然存在用户身份被盗用的问题，这个问题实际上很难完 美解决，我们能做的，只能是当发生用户身份被盗用这样的事情时，将损失降低到最小。 因此，我们来看下另一种方案，就是二次校验。 二次校验这块，实现起来要稍微复杂一点，我先来和大家说说思路。 为了让用户使用方便，我们开通了自动登录功能，但是自动登录功能又带来了安全风险，一个规避的办 法就是如果用户使用了自动登录功能，我们可以只让他做一些常规的不敏感操作，例如数据浏览、查 看，但是不允许他做任何修改、删除操作，如果用户点击了修改、删除按钮，我们可以跳转回登录页 面，让用户重新输入密码确认身份，然后再允许他执行敏感操作。

也就是说一些敏感的操作，还是需要用户进行登录才能访问的到。

具体的操作很简单，

.antMatchers(“/rememberme”).rememberMe() –>表示自动登录即可访问

.antMatchers(“/admin”).fullyAuthenticated() –>表示需要输入用户名密码才能访问的到
