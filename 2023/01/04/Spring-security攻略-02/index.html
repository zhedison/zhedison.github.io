Spring security攻略-02

一、密码加密

1、PasswordEncoder介绍

在Spring Security中与密码加密相关的都是由PasswordEncoder来主导的，这个接口有很多是实现类

其中最常用的就是第一个BCryptPasswordEncoder了，PasswordEncoder接口只有三个方法：

  • encode：用于对密码进行加密
  • matches：用于对密码进行比对
  • upgradeEncoding：用于表示密码是否需要再次加密，比如以前的密码用的是md5加密，但是现在md5加密不安全了，想换成别的加密方式，那么就需要对密码进行更新操作。

那么这里面进行密码校验的PasswordEncoder又是怎么来的呢？

3、默认的PasswordEncoder

我们知道如果我们自己没有配置PasswordEncoder的话会默认创建一个PasswordEncoder，那什么是默认的？

看看PasswordEncoderFactories里面的这个方法

返回了一个叫做DelegatingPasswordEncoder的PasswordEncoder，看这个名字像是一个代理的PasswordEncoder，看看他的定义

解析：

  1. DelegatingPasswordEncoder 也是实现了 PasswordEncoder 接口，所以它里边的核心方法也是 两个：encode 方法用来对密码进行编码，matches 方法用来校验密码。
  2. 在 DelegatingPasswordEncoder 的构造方法中，通过 通过传入的两个参数 encodingId 和 encoders ，获取到默认的编码器赋值给 passwordEncoderForEncode，默认的编码器实际上就 是 BCryptPasswordEncoder。
  3. 在 encode 方法中对密码进行编码，但是编码的方式加了前缀，前缀是 {编码器名称} ，例如如果 你使用 BCryptPasswordEncoder 进行编码，那么生成的密码就类似 {bcrypt}$2a$10$oE39aG10kB/rFu2vQeCJTu/V/v4n6DRR0f8WyXRiAYvBpmadoOBE. 。这样有什 么用呢？每种密码加密之后，都会加上一个前缀，这样看到前缀，就知道该密文是使用哪个编码器 生成的了。
  4. 最后 matches 方法的逻辑就很清晰了，先从密文中提取出来前缀，再根据前缀找到对应的 PasswordEncoder，然后再调用 PasswordEncoder 的 matches 方法进行密码比对。
  5. 如果根据提取出来的前缀，找不到对应的 PasswordEncoder，那么就会调用 UnmappedIdPasswordEncoder#matches 方法，进行密码比对，该方法实际上并不会进行密码 比对，而是直接抛出异常。

4、密码加密自动升级（其实是根据你的配置来进行改变）

Spring Security 支持根据你配置的PasswordEncoder来进行密码加密方式的改变，比如说之前是明文的方式存储的，前缀是{noop}，现在要进行bcrypt加密，那一大堆密码在数据库里面，不可能一个个的去改，而且加密过后的密码（比如之前是md5加密）也改不了。解决方法就是等用户再次登录的时候，根据用户登录时输入的密码来进行加密，然后根据用户名来去数据库更新。注意：PasswordEncoder只能用DelegatingPasswordEncoder这个实例，因为只有这个实例是可以识别前缀的。
